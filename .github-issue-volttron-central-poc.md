## Summary

Lab work on **VOLTTRON Central** (via [volttron-docker](https://github.com/VOLTTRON/volttron-docker)) plus **Open-FDD** (`./scripts/bootstrap.sh` → `afdd_stack/scripts/bootstrap.sh`) surfaced **gaps in docs, operator mental models, and bootstrap alignment**. This issue captures what **worked**, where we **fell short**, and the **long-haul goal**: a **turnkey** path where **Central + `afdd_stack` / `openfdd_stack`** are up reliably, and **external agents (e.g. Open Claw)** can drive **edge → Central → Open-FDD** workflows without ambiguous host-vs-container steps.

## Context (reference layout)

| Piece | Role |
|--------|------|
| **Repo root** `./scripts/bootstrap.sh` | Thin wrapper → `afdd_stack/scripts/bootstrap.sh` |
| **`afdd_stack/scripts/bootstrap.sh`** | `--doctor`, `--central-lab`, `--compose-db`, `--volttron-docker`, UI build flags, `VOLTTRON_HOME` stub/quarantine, tests, etc. |
| **`afdd_stack/openfdd_stack/`** | Platform SQL/FDD loop, `volttron_bridge`, etc. — **not** started by `volttron-docker`; brought in via **compose-db**, FastAPI-from-source, or agents as documented |
| **`./scripts/volttron-docker.sh`** | Runs `docker compose` only under **`OFDD_VOLTTRON_DOCKER_DIR`** (default `~/volttron-docker`) — **Open-FDD does not commit there** |

## What worked (PoC, lab)

- **Central in Docker (`volttron1`)**: platform healthy; **`vctl status`** shows driver, historian, actuator, listener, **VCP**, **VC**.
- **Web**: **`https://<LAN>:8443/vc/index.html`** and **`/admin/login.html`** return **200** — correct entrypoints; bare **`/vc/`** may **404** (routing quirk, not “Central dead”).
- **Fast PoC rule**: treat **container `VOLTTRON_HOME`** (e.g. `/home/volttron/.volttron`) as **source of truth**; run **`vctl`** via **`docker exec -u volttron volttron1 bash -lc '…'`**.

## Where we fell short (document + product gaps)

### 1. `VOLTTRON_HOME` / `--central-lab` mismatch

- **`./scripts/bootstrap.sh --central-lab`** writes stubs under **host** `$VOLTTRON_HOME` (often `~/.volttron`).
- Default **upstream** `docker-compose.yml` may **only** mount `./configs` and use **internal** container `VOLTTRON_HOME`, so **host stubs are not the live runtime tree** unless compose **bind-mounts** a host dir per [volttron-docker README](https://github.com/VOLTTRON/volttron-docker).
- **Impact**: operators chase “duplicate `web-ssl-cert`” / config fixes on the **wrong filesystem**; confusion for humans and **Open Claw**.

**Doc fix (partially landed):** `docs/howto/volttron_central_and_parity.md` — “Compose vs host stubs” + Central UI probe notes. **Still needed:** a single **“choose one”** runbook box: *PoC internal home* vs *persistent bind mount* with exact compose snippet pointers (upstream-owned file).

### 2. Central UI entry vs probes

- Operators tested **`/vc/`** and concluded Central was broken; **upstream** documents **`/vc/index.html`**.
- **Doc fix:** `afdd_stack/volttron_agents/openfdd_central_ui/README.md` + parity doc — clarify **`/vc/index.html`** / **`/admin/login.html`**.

### 3. Edge (native Pi) ↔ Central registration

- **Central** on LAN (e.g. `192.168.204.16:8443`) is healthy; **edge** (e.g. `192.168.204.12` native `~/volttron`) may run agents but **without** `volttron-central-address`, **VCP**, or Central approval — **`vctl auth remote list`** empty.
- **Gap:** one short **Open-FDD doc page** (or section) that maps **“native edge checklist”** ↔ **`docker exec` on Central**, with explicit **LAN URLs** (not `127.0.0.1` from another host).

### 4. Admin / Central login credentials

- No repo-stored default; first-time **`/admin/login.html`** vs existing **`web-users`** under live `VOLTTRON_HOME` needs a **one-paragraph** “who set the password” note in **Getting started** or parity doc (link upstream reset guidance only).

### 5. `openfdd_central_ui` / `/openfdd/` optional layer

- PoC skipped **Open-FDD static UI on Central**; path requires **build + `agent-config.json` + `pip install -e` + `vctl install` inside container**, often needing **repo bind mount** or `docker cp`.
- **Gap:** smallest **documented** path from monorepo root when **`volttron1` has no host checkout mounted**.

### 6. Open Claw / AI-driven ops

- Agents need a **pinned default**: container canonical for Central-on-Docker; native `~/volttron` on edge Pis; **never** mix without stating shell context.
- **Gap:** optional **`docs/operations/`** short **“Open Claw lab prompt”** or link to `docs/openclaw_integration.md` §1f + **execution-context** one-liner (could live in repo or stay external).

### 7. `afdd_stack` “full stack” vs current default path

- README / GitHub landing still mentions **`./afdd_stack/scripts/bootstrap.sh`** in places; canonical entry is **repo root** `./scripts/bootstrap.sh`.
- **`--central-lab`** brings **Timescale + stubs + volttron-docker clone** — it does **not** by itself start **legacy all-in-one** `afdd_stack/stack` Caddy/API/React compose (see `afdd_stack/legacy/README.md` / security docs).
- **Gap:** explicit **matrix** in **Getting started**: “I want … → run these flags” for (a) SQL only, (b) Central lab, (c) full FastAPI+UI stack from compose, (d) engine-only.

## Long-haul goals (acceptance-oriented)

1. **VOLTTRON Central “100%” for reference lab** — Documented: compose up, **correct web URLs**, admin bootstrap, **remote auth approve**, optional **`openfdd_central_ui`**, known-good **`curl`** probes.
2. **`afdd_stack` / `openfdd_stack` up in one coherent bootstrap story** — Clear flags for DB, env files, optional API/UI (where supported), and how that **coexists** with **volttron-docker** (no duplicate ownership of the same DB).
3. **Turnkey edge onboarding** — From zero: edge **`vcfg`** answers (ZMQ, HTTPS Central URL), **VCP**, Central **`vctl auth remote approve`**, visible in **`/vc/index.html`**.
4. **Non-VOLTTRON / “other” instances** — Document boundaries: what must be VOLTTRON vs what can be **HTTP-only** to Open-FDD; how **Open Claw** uses **`OFDD_API_KEY`** + SSH (per `docs/openclaw_integration.md`).
5. **AI-driven workflows** — Repeatable prompts, **no secrets in chat**, **container vs host** explicit.
